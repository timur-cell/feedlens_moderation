FeedLens Moderation — Architecture

FeedLens is JamesEdition's internal listing-moderation tool. Listings arrive from the LAS pipeline (webhook push) or are fetched on demand from the JE mobile API, then flow through a deterministic rule engine augmented with optional AI — Claude vision on listing images, an AI parameter scan, and LLM verification of borderline matches. Decisions are persisted, optionally forwarded to Implio (the third-party moderation platform), and surfaced in a React SPA where moderators work the manual queue, override outcomes, and manage rules, keyword lists, message templates and team accounts. Originally built on Convex/TypeScript (frozen in convex/, branch typescript-v1), the backend was fully rewritten to Rails 8 + PostgreSQL with the SPA kept intact — engine parity is proven by golden fixtures generated by executing the original TypeScript evaluators.

1System overview

Everything runs as a five-service Docker Compose stack. The browser talks to nginx (which serves the SPA build and proxies /api and /image-proxy to Rails); in production Caddy fronts the stack with automatic Let's Encrypt TLS. Rails (Puma behind Thruster) and the Solid Queue worker share one PostgreSQL instance holding the app database and a separate queue database.

  • Same-origin by design — nginx proxies /api to Rails so Devise session cookies and the CSRF token never cross origins. There is no CORS for the SPA.
  • Two unauthenticated entry points only: POST /api/push-flagged (API-key auth for LAS) and GET /image-proxy (strict jamesedition.com hostname allowlist, https-only, parsed-hostname check to prevent SSRF).
  • AI is optional — without ANTHROPIC_API_KEY the engine still runs; AI-trigger matches simply route to the manual queue. Implio submission is stubbed (payload logged, no HTTP) unless IMPLIO_STUB=false.

2The moderation pipeline

Source of truth: rails/app/services/moderation/runner.rb (the orchestrator, a port of the Convex moderateListing action) and rails/app/services/listings/fetch_and_moderate.rb (the fetch-then-moderate flow behind Moderate by URL).

Step by step

  1. 0 — AI parameter scan (Ai::ParamScan): deterministic threshold pre-checks (price/sqm anomalies, area mismatches, missing critical data, …) merged with a Claude contextual verdict (reject | review | ok), persisted as an ai_parameter_scans row. Cached — a previous scan is reused unless force-rescanned. Failures are logged and never affect the moderation outcome.
  2. 1 — Load configuration: all enabled rules, sorted by priority ascending (stable tie-break on insertion order, matching JS Array#sort); moderation lists resolved into rule configs via listRef/additionalListRef/excludeListRef; settings (single merged row over defaults).
  3. 2 — Engine#evaluate: pure-Ruby deterministic evaluation (no Rails, no ActiveRecord — see §3). Produces either a final outcome or needs_llm when an auto_ai/former_manual rule matched.
  4. 3 — On-demand vision: if needs_llm and the listing has image_urls but no vision data (chat_gpt_property_condition is nil), Ai::VisionAnalyzer scores up to maxImagesPerVisionScan images (setting, default 10), the chat_gpt_* fields are patched onto the listing, and hybrid_vision rules are re-evaluated once. Only a freshly matched auto/reject hybrid rule may short-circuit to rejection here.
  5. 4 — LLM verification (Moderation::LlmVerifier, a port of the TS callLlm): Claude reviews the AI-trigger matches and returns recommendation/confidence/notice/assessment. Engine#decide_with_llm applies the autoApproveThreshold (default 0.9) and autoRejectThreshold (default 0.85) from settings; anything below threshold, any invalid response, or a missing API key routes to the manual queue.
  6. 5 — Persist: a ModerationResult row (camelCase rule_matches jsonb, llm_response, confidence, epoch-ms processed_at), the listing's moderation_status, and match_count/last_matched_at on every matched rule (the synthetic llm_assessment match is excluded).
  7. 6 — Implio submission (Integrations::ImplioClient): forwards the decision so existing Implio rules act on JamesEdition. With IMPLIO_STUB unset or "true" (the default) the full payload is logged and no HTTP request is made. Errors are swallowed — Implio can never fail a moderation run.

3Rule engine

Moderation::Engine is pure Ruby (no Rails, no ActiveRecord) operating on camelCase hashes — the exact shape of the Convex documents and of rails/db/seed_data/*.json. Every rule has two independent axes: a category (which evaluator runs, and whether it is deterministic or AI-triggering) and a tier + action (what a match means for the outcome).

Categories (which evaluator, when)

CategoryPhaseEvaluator & behavior
simple_codedeterministic Field conditions with operators (below), plus optional guards: countryFilter, office/account-type filters, feedSource, minPrice. Rules with patterns/textLists dispatch to the regex/keyword evaluator (lists resolved via listRef/excludeListRef); rules with officeIds/officeNames dispatch to the office evaluator.
hybrid_visiondeterministic Checks against the chat_gpt_* vision scores (property condition, watermark share/text, image quality/type). Skipped entirely when the listing has no vision data.
internaldeterministic Structural rules (e.g. the synthetic data_fetch_failed); evaluated alongside simple_code.
auto_ai · former_manualAI-trigger Evaluated after the deterministic pass (simple or regex evaluator). A match sets needs_llm → on-demand vision (step 3) and LLM verification (step 4).
accuracydisabled Accuracy rules exist in seed data but are excluded from the deterministic filter — the TS version commented them out on 2026-03-17 and the Ruby port mirrors this exactly.

Tier × action (what a match means)

TierMeaning
autoDeterministic decision: reject rejects immediately, approve approves immediately, notice approves-with-notice if nothing stronger matched.
manual / verifyA match (or any action: flag) sends the listing to the human review queue.

Condition operators

From rails/app/services/moderation/condition_evaluator.rb — exact JS semantics: numeric operators coerce both sides with JS Number() (NaN compares false), string operators compare String() values case-insensitively, and a nil listing value only ever satisfies empty/is_null (missing data never matches).

lt lte gt gte < <= > >= eq neq == != in not_in contains not_contains matches empty not_empty is_null is_not_null is_true is_false

matches builds a case-insensitive RegExp and returns false on an invalid pattern rather than raising.

Country normalization quirks

country_matcher.rb reconciles the three spellings JE data uses: ISO codes in rules (ES, IT…), full names in listings ("Spain", "United Kingdom"), and sometimes region names ("Algarve" → PT, "Tuscany" → IT, "Dubai" → AE, "Florida" → US). UK is normalized to the real ISO code GB so filters using either spelling match either form. Matching tries: direct/case-insensitive equality → name→code → code→canonical-name → region fallback.

Golden-fixture parity

How parity with the TypeScript original is proven. scripts/golden/generate.ts (run with bun) extracts the pure prelude of convex/moderation.ts — evaluateCondition, evaluateSimpleRule, evaluateRegexRule, evaluateHybridVisionRule, evaluateOfficeRule, resolveListRefs, the country maps — without rewriting any logic, loads the real seed rules/lists plus synthetic x_-prefixed extras that exercise edge paths (full operator set, hybrid check types, office rules, minPrice, priority ties, a disabled rule), executes every fixture through the original decision flow, and writes rails/spec/fixtures/golden/expected.json. The result: 66 golden fixtures, 59 distinct rules fired, asserted in Ruby per fixture on ordered match names, outcome, needs-LLM flag, LLM rule matches and seller message — all green.

4Data model

14 tables (rails/db/migrate/, one migration per table). Two conventions inherited from the Convex era: domain timestamps are epoch-millisecond bigints (imported_at, processed_at, scanned_at, created_at_ms, …) alongside Rails' own created_at/updated_at datetimes, and jsonb payloads keep camelCase keys (rule_matches, raw_data, llm_response…) so the SPA consumes records in the original Convex document shape.

  • Natural keys: listings.je_id, rules.name, moderation_lists.name, message_templates.name, moderators.email, daily_stats.date, settings.key are unique — these drive the idempotent convex:import upserts (§7).
  • settings is one row, merged over Setting.defaults at read time (Setting.current) — nil columns fall back to the defaults ported verbatim from convex/settings.ts.
  • listings.chat_gpt_* are the persisted Claude vision scores (property condition, watermark share/text, image quality/type, conclusion) that hybrid_vision rules read.

5REST API

Full contract with payload shapes: docs/rails_api_contract.md. Everything is JSON under /api; records serialize like Convex docs (_id, _creationTime, camelCase keys, epoch-ms timestamps), so SPA components consume them unchanged.

Auth model. Devise session cookie + CSRF token: the SPA calls GET /api/session to get { user, csrfToken } (also set as a readable XSRF-TOKEN cookie) and sends X-CSRF-Token on every non-GET request. Roles: viewer (read), moderator (moderate & override), admin (manage rules / lists / messages / settings / users). Any active account passes the moderator check (viewers included — TS parity); admin-only writes return 403 otherwise.

Data layer

  • src/lib/apiClient.ts — typed REST client; Convex-style args objects keep call sites mechanical; injects the CSRF token on non-GET requests.
  • src/hooks/useApiQuery.ts — polling replacement for Convex's reactive useQuery/useMutation: fetch on mount/arg change, silent pollMs polling (sidebar badge 7 s, queue 10 s), and a global refetch registry — useApiMutation refetches all mounted queries after a successful mutation, restoring cross-component reactivity.
  • src/contexts/AuthContext.tsx — probes GET /api/session on boot, stores user + CSRF token, exposes signIn/signOut; any API 401 dispatches a global auth-expired event that drops the session.

Importing production data — convex:import

bin/rails "convex:import[/path/to/export.zip]" (rails/lib/tasks/convex_import.rake) imports an npx convex export ZIP or directory (both layouts supported). It maps camelCase docs to snake_case columns, remaps Convex _id references to integer FKs (falling back to je_id lookup), and upserts on natural keys — composite keys for history tables — so re-runs are idempotent. Semantics are a merge: prod rows win on natural-key conflicts while seed-only rows remain. Verified against the real production export (June 2026): 67 listings, 75 results, 82 rules after merge, 6 moderators — zero errors, clean second run. Convex Scrypt password hashes are not portable; imported moderators get a random password and use reset / admin set-password.
